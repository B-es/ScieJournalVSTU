# Техническая спецификация (TS)

---

## 1. Обзор

**Тип проекта:** Многостраничное веб-приложение (SPA-фронтенд на Vue3 + серверный REST API на Django) с публичной Open Access частью (индексируемой поисковиками) и личными кабинетами (SPA за авторизацией) для четырёх ролей из PRD: Автор, Рецензент, Главный редактор, Технический редактор.

**Ключевые технические ограничения:**
- Публичные страницы (главная, архив, статья, поиск) должны хорошо индексироваться поисковыми системами и научными базами (SEO из раздела 6 PRD) — для SPA на Vue3 это требует SSR/пререндеринга (Nuxt3 или prerendering) хотя бы для публичной части.
- Бюджет хостинга ограничен одним VPS на старте — архитектура должна быть монолитной/лёгкой в развёртывании, без отдельного кластера сервисов.
- Существующей инфраструктуры у вуза для проекта нет, разворачиваем с нуля.
- Двуязычность (RU/EN) должна поддерживаться на уровне и фронтенда, и данных (модель данных — раздел 6), а не только UI-строк.
- Загрузка и хранение файлов (рукописи, рецензии, документы, PDF выпусков) — потенциально большие объёмы на диске VPS, нужна оценка роста хранилища.
- Форма подачи статьи поддерживает сохранение как черновика (US-11 PRD) — требует автосохранения на фронтенде (дебаунс-запросы к `/api/articles/draft`) без обязательной серверной валидации полей до финальной отправки.

## 2. Технологический стек

| Слой | Технология | Почему выбрано |
| ---------------- | ----------------------------- | ----------------------------------------------------------------------------------------------- |
| Frontend | Vue3 (+ Vue Router, Pinia) | Реактивность и простота, быстрый вход в разработку, хорошо подходит для личных кабинетов (SPA) |
| Публичные страницы | Vue3 с SSR/пререндерингом (Nuxt3) | Нужна индексация статей и выпусков поисковиками (SEO из раздела 6 PRD), чистый SPA для этого не подходит |
| Backend / API | Django + Django REST Framework (DRF) | Много готовых модулей (auth, admin, ORM), быстрее всего настроить бизнес-логику процесса рецензирования |
| Админ-панель (Технический редактор) | Django Admin (кастомизированная) | Закрывает раздел 5 PRD «Административная панель» без отдельной разработки UI — управление пользователями, выпусками, настройками |
| База данных | PostgreSQL (в разработке — подмена на SQLite) | Надёжное реляционное хранилище со связями Статья–Версии–Рецензии–Выпуски; SQLite — только для локальной разработки/тестов |
| Хостинг / деплой | VPS (Nginx + Gunicorn/Uvicorn + Docker Compose) | Соответствует ограничению бюджета из раздела 1; Docker Compose упрощает деплой монолита из нескольких сервисов на одном сервере |
| CMS | Django Admin | Технический редактор администрирует пользователей, выпуски и настройки журнала через кастомизированную Django Admin, не требуя отдельной CMS |
| Файловое хранилище | Локальная ФС VPS через Django (media/), с возможностью перехода на S3-совместимое хранилище | Простое решение для MVP в рамках одного VPS; абстракция Django storages позволяет позже перейти на облако без изменения бизнес-логики |
| Email-рассылка | Django email backend + SMTP-провайдер (например, Yandex/SendGrid — уточняется отдельно) | Обязательное требование из раздела 3 PRD «Границы интеграций» — уведомления о статусах, приглашениях, решениях |
| DOI | Внешний провайдер DOI (CrossRef/НЭИКОН — уточняется отдельно), интеграция через REST API провайдера | Обязательное требование из раздела 5 PRD «Техническая подготовка и публикация» |

## 4. Стандарты кода

- Язык/версия: TypeScript 5.x (frontend), Python 3.12 (backend)
- Линтер / форматтер: ESLint + Prettier (frontend), Ruff + Black (backend)
- Соглашение об именовании:
	- TS: camelCase для переменных и функций, PascalCase для компонентов Vue и типов
	- PY: snake_case для переменных и функций, PascalCase для классов и моделей Django
- Структура коммитов: Conventional Commits (`feat:`, `fix:`, `chore:`, `docs:` и т.д.)
- Код-ревью: не обязательно (командой принято решение не блокировать деплой ревью, но рекомендуется для модулей рецензирования и публикации как наиболее критичных для бизнес-логики)

## 5. Архитектура

**Архитектурный паттерн:** клиент-серверный монолит на старте — единый Django-бэкенд с REST API (DRF) обслуживает и Vue3/Nuxt3-фронтенд, и Django Admin для технического редактора. Разделение на микросервисы не требуется при текущем масштабе (один вуз, MVP).

**Схема взаимодействия компонентов:**
```
Читатель / Автор / Рецензент / Гл. редактор
        │  (браузер)
        ▼
   Vue3 / Nuxt3 SPA  ──REST/JSON (DRF)──►  Django Backend
   (публичные страницы + личные кабинеты)        │
                                                   ├─► PostgreSQL (данные: статьи, рецензии, выпуски, пользователи)
                                                   ├─► media/ или S3 (файлы: рукописи, рецензии, PDF)
                                                   ├─► SMTP (email-уведомления)
                                                   └─► внешний DOI-провайдер (REST API)

Технический редактор
        │  (браузер)
        ▼
   Django Admin ──ORM──► PostgreSQL (та же БД, что и у REST API)
```

**Потоки данных (примеры по ключевым US из PRD раздела 4):**
- US-1: Автор → форма подачи (Vue) → `POST /api/articles` → валидация на бэкенде → запись в БД (статус «На рассмотрении») → email-уведомление техническому редактору.
- US-4: Главный редактор → форма назначения рецензентов (Vue) → `POST /api/articles/{id}/reviewers` → создание записей `Review` со статусом «приглашён» → email-уведомления рецензентам.
- US-7: Главный редактор → форма решения (Vue) → `POST /api/articles/{id}/decision` → обновление статуса статьи → email-уведомление автору и рецензентам.
- US-9: Технический редактор (Django Admin) → присвоение DOI и публикация → запрос к внешнему DOI-провайдеру → обновление статуса статьи на «Опубликована» → появление статьи в публичном API `/api/public/articles`.

## 7. API

### Модуль: Аутентификация

| Метод | Путь | Назначение | Тело запроса | Ответ | Коды ошибок |
|---|---|---|---|---|---|
| POST | /api/auth/register | регистрация (роль «Автор» по умолчанию) | `{fullName, email, password}` | `{userId, token}` | 400, 409 |
| POST | /api/auth/login | вход | `{email, password}` | `{token, roles}` | 400, 401 |
| POST | /api/auth/logout | выход | — | `{status}` | 401 |
| POST | /api/auth/password-reset | восстановление пароля | `{email}` | `{status}` | 400, 404 |

### Модуль: Статьи (Автор) — US-1, US-3, US-8, US-11

| Метод | Путь | Назначение | Тело запроса | Ответ | Коды ошибок |
|---|---|---|---|---|---|
| POST | /api/articles/draft | создать/обновить черновик статьи (US-11), без обязательной валидации полей | `{titleRu?, titleEn?, abstractRu?, abstractEn?, keywordsRu?, keywordsEn?, authors?, manuscriptFile?, documents?}` | `{articleId, status: "draft", lastAutosavedAt}` | 401 |
| POST | /api/articles/{id}/submit | финальная отправка статьи (черновик → «На рассмотрении», US-1) | `{titleRu, titleEn, abstractRu, abstractEn, keywordsRu, keywordsEn, authors, manuscriptFile, documents}` | `{articleId, status}` | 400, 401, 403, 422 |
| GET | /api/articles | список своих статей с фильтром по статусу (включая черновики) | query: `status` | `{items: [...]}` | 401 |
| GET | /api/articles/{id} | карточка статьи с историей статусов и рецензиями | — | `{article, versions, reviews, decisions}` | 401, 403, 404 |
| POST | /api/articles/{id}/versions | загрузить исправленную версию (US-3, US-8) | `{manuscriptFile, documents, authorComment}` | `{versionId, status}` | 400, 401, 403, 422 |

### Модуль: Проверка комплектности (Технический редактор) — US-2

| Метод | Путь | Назначение | Тело запроса | Ответ | Коды ошибок |
|---|---|---|---|---|---|
| GET | /api/articles?status=submitted | очередь статей на проверку комплектности | — | `{items: [...]}` | 401, 403 |
| POST | /api/articles/{id}/completeness-check | результат проверки комплектности | `{approved: bool, comment}` | `{status}` | 400, 401, 403, 404 |

### Модуль: Предварительная проверка и рецензенты (Главный редактор) — US-4, US-5

| Метод | Путь | Назначение | Тело запроса | Ответ | Коды ошибок |
|---|---|---|---|---|---|
| POST | /api/articles/{id}/topic-check | решение о соответствии тематике | `{approved: bool, comment}` | `{status}` | 400, 401, 403, 404 |
| POST | /api/articles/{id}/reviewers | назначить рецензентов (мин. 2) | `{reviewerIds: [...], deadline}` | `{reviews: [...]}` | 400, 401, 403, 422 |
| POST | /api/reviews/{id}/reassign | назначить замену отказавшемуся рецензенту | `{newReviewerId, deadline}` | `{reviewId, status}` | 400, 401, 403, 404 |

### Модуль: Рецензирование (Рецензент) — US-5, US-6

| Метод | Путь | Назначение | Тело запроса | Ответ | Коды ошибок |
|---|---|---|---|---|---|

## 8. Аутентификация и авторизация

- Метод аутентификации: email + пароль (регистрация из DS «Регистрация» — ФИО, email, пароль); вход по email + паролю со страницы «Вход».
- Хранение сессии: JWT (access + refresh токен) для Vue3-фронтенда и публичного/личного API; для Django Admin (Технический редактор) — стандартная cookie-сессия Django.
- Роли и права: пользователь может иметь несколько ролей одновременно (например, быть и автором, и рецензентом); доступные разделы личного кабинета определяются набором ролей пользователя, в соответствии с разделом 2 PRD.

| Роль | Доступные действия |
|---|---|
| Гость (не авторизован) | просмотр публичной части: главная, архив, поиск, страница статьи (US-10), регистрация, вход |
| Автор | + подача статьи, просмотр своих статей и статусов, загрузка исправленных версий, просмотр рецензий и комментариев по своим статьям (US-1, US-3, US-8) |
| Рецензент | + просмотр приглашений и назначенных статей, принятие/отказ от приглашения, доступ к полному тексту после согласия, заполнение и отправка рецензии (US-5, US-6) |
| Главный редактор | + просмотр всех поступивших статей, проверка соответствия тематике, назначение/замена рецензентов, просмотр всех рецензий, принятие редакционного решения (US-4, US-7) |
| Технический редактор (через Django Admin) | + проверка комплектности материалов, управление пользователями и ролями, присвоение DOI, формирование выпусков, публикация статей, настройки журнала (US-2, US-9) |

## 9. Управление состоянием

- Инструмент: Pinia (официальное решение для Vue3).
- Глобально хранится:
  - `authStore` — текущий пользователь, токен, набор ролей, выбранный язык интерфейса (используется во всех layout'ах — публичном и личного кабинета, см. DS раздел 1).
  - `notificationsStore` — счётчик непрочитанных уведомлений (отображается в верхнем меню личного кабинета согласно карте сайта DS).
  - `uiStore` — состояние языка (RU/EN) для i18n, глобальные флаги загрузки/ошибок для тостов.
- Локально в компонентах хранится:
  - состояние конкретных форм (подача статьи, форма рецензии, форма редакционного решения) — до момента отправки на бэкенд;
  - состояние таблиц/фильтров списков («Мои статьи», «Найденные статьи») — фильтры, пагинация, сортировка;
  - состояние отдельных виджетов (раскрытие карточки, состояние PDF-viewer на странице статьи).

## 10. Обработка ошибок и логирование

- Формат ошибок API: единый JSON-формат `{code, message, fieldErrors?}`, где `fieldErrors` используется для инлайн-ошибок валидации форм (подача статьи, рецензия и т.д.).
- Отображение ошибок пользователю: тосты — для общих/сетевых ошибок и подтверждений действий; инлайн-сообщения под полями формы — для ошибок валидации (соответствует состоянию «ошибка» из раздела 6 DS).
- Логирование: серверные логи и исключения — Sentry; логи доступа/деплоя — консоль сервера (журналы Nginx/Gunicorn на VPS).
- Уровни логов: error (исключения, сбои интеграций с DOI-провайдером и email), warn (неудачные попытки входа, истёкшие дедлайны рецензентов), info (ключевые бизнес-события: подача статьи, смена статуса, публикация), debug (только в окружении разработки).

## 11. Производительность

*Цели соответствуют разделу 6 PRD.*

- Целевой Lighthouse Performance score: ≥ 90 для публичных страниц (главная, архив, статья, поиск).
- Целевое время загрузки: TTFB ≤ 600 мс, LCP ≤ 2.5 сек, CLS ≤ 0.1 на публичных страницах при стабильном соединении.
- Стратегия кеширования: SSR-кеш/пререндер публичных страниц (Nuxt3), кеш браузера для статики (шрифты DM Sans, иконки Lucide, изображения выпусков), кеш API-ответов для списков опубликованных статей и выпусков (редко меняются).
- Оптимизация изображений: форматы WebP/AVIF для обложек выпусков и изображений статей, lazy loading для карточек в списках («Компактная карточка статьи», «Статья»), responsive images под breakpoints из DS раздела 7.

## 13. Тестирование

| Тип теста | Инструмент | Что покрывает |
|---|---|---|
| Unit | Vitest (frontend), pytest (backend) | утилиты форматирования, валидаторы форм, бизнес-логика статусной модели статьи |
